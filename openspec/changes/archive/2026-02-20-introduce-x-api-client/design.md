## Context

現在のX API呼び出しは各コマンド内のスタブ実装に分散しており、HTTPリクエスト/認証/エラー分類が共通化されていません。後続の実API化では、同一の認証・エラー処理・JSON変換を再利用できる基盤が必要です。

## Goals / Non-Goals

**Goals:**
- 同期型のHTTPクライアントを用意し、ベースURL・認証ヘッダー・共通ヘッダーを統一する
- 429/401/403/5xx などを既存のエラー表現に分類する
- モック/テストでHTTPリクエスト検証ができる構成にする

**Non-Goals:**
- 各エンドポイント（tweets/search/timeline/media）の実装はこの変更では行わない
- ストリーミングAPIや非同期I/Oの導入は行わない

## Decisions

- **同期HTTPクライアントを採用**: 既存CLIは同期処理が前提のため、`ureq` などの同期HTTP依存を利用する。非同期ランタイムの導入は避ける。
- **ベースURLを設定可能にする**: `XCOM_RS_API_BASE` で切替可能にし、モックサーバ（mockito）でも検証できるようにする。
- **エラー分類を共通化**: レスポンスのHTTPステータス/ヘッダーから `ClassifiedError` 相当へ変換し、`retryAfterMs` を抽出する。
- **クライアント抽象を提供**: `XApiClient` トレイト + `HttpXApiClient` 実装で、実装とテストを分離する。

## Risks / Trade-offs

- [HTTP依存の追加] → 依存は同期最小構成に限定し、API使用箇所は共通モジュールのみに集中させる
- [APIレスポンス変更] → 失敗時は詳細ログとフォールバックのエラーメッセージを提供する
- [レート制限情報の欠落] → `retry-after` が無い場合は既定値で扱う
