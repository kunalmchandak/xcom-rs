# refactor-tweets-commands 提案

## なぜ
- `src/tweets/commands.rs` が巨大で、引数定義・実装・テストが同居しているため見通しが悪い。
- 機能追加時に同一ファイルの編集が集中し、衝突やレビュー負荷が増える。

## 何を変えるか
- `tweets` コマンドの引数・実装・テストを機能単位で分割する。
- 既存の公開 API と挙動は維持し、内部構造のみを整理する。

### 対象範囲
- `tweets` のコマンド実装・テストのモジュール分割
- 既存テストの再配置と回帰テストの追加

### 非対象
- CLI の引数や出力仕様の変更
- API の新規追加

### 成果物
- 機能別モジュール（create/list/engagement/thread など）
- 主要フローの回帰テスト

### 受け入れ条件
- 既存の `tweets` 系コマンドの出力が変更されないこと
- `cargo test` と `cargo clippy -- -D warnings` が通ること

### リスク・留意点
- 既存のテスト構成変更に伴うモジュール可視性の調整が必要になる
