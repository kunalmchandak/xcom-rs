# refactor-introspection-registry 提案

## なぜ
- `src/introspection.rs` が 1,700 行超の単一ファイルで、コマンド一覧・スキーマ・ヘルプが同居しており拡張時の編集範囲が広い。
- コマンド追加時に複数箇所を更新する必要があり、記述の不一致や漏れのリスクがある。

## 何を変えるか
- コマンドメタデータの「登録元」を単一化し、一覧/スキーマ/ヘルプ生成が同じ定義を参照する構成に分割する。
- `introspection` を小さなモジュールに分割し、責務を明確化する。

### 対象範囲
- `introspection` の内部構成とテストの整理
- 既存の出力（`commands` / `schema` / `help`）の非機能変更

### 非対象
- CLI の公開仕様変更
- 出力フォーマット・データ構造の変更

### 成果物
- データ駆動のコマンド登録レイヤー
- 分割されたモジュール構成と回帰テスト

### 受け入れ条件
- 既存の `commands` / `schema` / `help` 出力が変更されないこと
- `cargo test` と `cargo clippy -- -D warnings` が通ること
- パフォーマンスの顕著な劣化がないこと

### リスク・留意点
- 生成元の共通化により、既存のテストが想定外の結合を検知する可能性がある
