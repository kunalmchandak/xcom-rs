# 変更提案: OAuth1.0aユーザー認証の追加

## 背景
Rust製の汎用CLIとして、OAuth2(PKCE)のuser contextだけでなくOAuth1.0a(3-legged)にも対応できる必要がある。現在はOAuth2の保存とBearerトークン解決が中心で、OAuth1.0aのログインフロー・保存・署名送信がない。

## 目的
- OAuth1.0aのログイン(手動/ローカルコールバック)を追加し、ユーザーアクセストークン/シークレットを保存できるようにする。
- API呼び出し時にOAuth1.0a署名ヘッダを選択できるようにし、OAuth2と並行運用できるようにする。

## スコープ
- OAuth1.0a 3-leggedフロー(リクエストトークン → 認可 → アクセストークン)の実装と保存
- 認証状態表示の拡張(authMode=oauth1a)
- OAuth1.0a署名ヘッダ生成とHTTPクライアントへの適用
- 既存OAuth2・Bearerトークンとの優先順位ルールの明確化
- モック/スタブによるテスト可能性の確保

## 非スコープ
- 既存APIの挙動変更(レスポンス形式の破壊的変更)
- OAuth2以外の新しい認可方式追加
- リアルな外部X API環境でのエンドツーエンド検証

## 成功基準
- `auth login` でOAuth1.0aを選択でき、認証情報が保存される。
- `auth status` がOAuth1.0aモードを表示できる。
- OAuth1.0aの認証情報がある場合、APIリクエストはOAuth署名ヘッダで送信できる。
- すべての主要APIクライアントがOAuth1.0a/2.0のいずれにも対応できる。

## 依存関係・リスク
- OAuth1.0a署名生成のライブラリ採用が必要(外部依存)。
- 既存の`auth.json`構造を壊さない後方互換が必要。
- OAuth1.0aは署名対象パラメータの扱いが複雑で、HTTPクライアント実装の揺れがリスク。
